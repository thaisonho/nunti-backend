# Phase 07.1: Production Infrastructure Enablement - Context

**Gathered:** 2026-04-09
**Status:** Ready for planning

<domain>
## Phase Boundary

Production AWS infrastructure setup focuses on establishing production-level networking, compute environment, database replication, and infrastructure components that go beyond the basic deployment foundation established in Phase 6. This phase ensures the production environment is properly isolated, monitored, and resilient to support live traffic.

This phase does NOT include:
- New product capabilities (messaging, keys, auth features)
- Advanced multi-region failover (single region with backup/restore only)
- Incident response runbooks (deferred to Phase 11)

</domain>

<decisions>
## Implementation Decisions

### Networking and VPC Isolation
- Production environment must use a separate VPC with distinct subnets from staging (not shared infrastructure).
- VPC must include NAT Gateways for outbound traffic from Lambda compute layer, enabling secure external API calls without exposing Lambda to inbound internet routing.
- Security groups should enforce least-privilege ingress/egress:
  - API Gateway: Accept HTTPS (443) from CloudFront distribution (not direct internet).
  - Lambda: Accept traffic from API Gateway only; egress to DynamoDB, Cognito, and external APIs as needed.
  - DynamoDB: Accept traffic from Lambda security group only (VPC endpoint or restricted ingress).
- API Gateway integration uses CloudFront CDN in front for caching, DDoS protection, and edge acceleration (cost-conscious on resilience features now).

### Database Resilience and Backup
- DynamoDB point-in-time recovery (PITR) enabled for all production tables.
- On-demand backups created daily with retention policy (minimum 7 days).
- DynamoDB global tables are NOT in scope (single region only; multi-region deferred to future).
- Provisioned throughput or on-demand billing decision: kept as "agent's discretion" — implement whichever aligns with cost-conscious strategy and projected traffic patterns.

### DNS and Routing
- Route53 is canonical for production DNS with health checks enabled on API Gateway endpoint.
- Health checks trigger automatic failover logic (implementation details: agent discretion on failover targets and routing policies).
- Staging continues to use its own Route53 record or direct API Gateway endpoint (no cross-environment DNS coupling).

### Cognito and Identity
- Separate production Cognito User Pool and App Client (full isolation from staging).
- Production User Pool must enforce stricter password/MFA policies than staging (if staging uses optional MFA, production requires it).
- Production User Pool default behavior must not leak auth config to staging or other environments.

### Cost vs. Resilience Trade-offs
- CloudFront CDN is included (edge caching + DDoS mitigation considered essential for production public endpoint).
- Spare capacity, multi-AZ RDS (not applicable to DynamoDB), and multi-region failover deferred to future release cycles.
- Infrastructure costs should be minimized while maintaining baseline resilience (single AZ acceptable, backups required).

### Claude's Discretion
- Exact provisioned throughput values or on-demand scaling policies for DynamoDB.
- Specific security group rule crafting and CIDR block management.
- CloudFront cache invalidation strategy and TTL policies.
- VPC endpoint configuration (whether to use VPC endpoints for DynamoDB or open security group ingress).
- Failover automation logic and Route53 routing policy specifics (weighted, latency-based, etc.).

</decisions>

<specifics>
## Specific Ideas

- Production infrastructure should be standalone and not share resource definitions with staging. Use separate parameter files, distinct stack names, and separate IAM roles.
- All production infrastructure should follow "least-privilege by default" — if a security group or IAM rule can be tighter, make it so. Err on the side of restrictive.
- The infrastructure in this phase enables the realtime reliability work in Phase 8 — ensure failover and health monitoring hooks are in place so Phase 8 can layer retry/reconnect logic.
- CloudFront + WAF are considered "resilience essentials"; multi-AZ compute and read replicas are "nice-to-have" (deferred).

</specifics>

<code_context>
## Existing Code Insights

### Reusable Assets
- `template.yaml`: Existing SAM template used for staging; will be parameterized to support separate VPC/security groups for production.
- `.github/workflows/release-deploy.yml`: Deployment automation that can be extended to deploy both staging and production stacks from the same manifest.
- `scripts/infra/setup-oidc-s3.sh`: OIDC bootstrap that establishes IAM roles; can be extended with production-specific role policies.
- `deploy/params.staging.json` and `deploy/params.production.json`: Existing environment parameter structure ready for production VPC/security group/Cognito references.

### Established Patterns
- Staging uses SAM + parameter-driven deployment (production follows the same pattern).
- Infrastructure as Code is the standard (no console-driven one-offs).
- Parameters are version-controlled; sensitive values injected at deploy time via GitHub Actions secrets.
- Immutable artifact promotion is used for rollback safety (production deployment reuses the same mechanism).

### Integration Points
- Production VPC will be created alongside staging, both managed by SAM template parameters.
- Production Cognito will have the same User Pool structure as staging but separate identity namespace.
- Route53 will serve both staging and production DNS, with separate health checks per environment.
- Production IAM roles will build on the OIDC setup from Phase 6, with additional service-specific permissions for VPC/security group management.

</code_context>
