Create scripts/deploy/backup-strategy.sh:

```bash
#!/bin/bash
set -e

# Backup Strategy: Daily DynamoDB backups with 7-day retention
# This script is invoked post-production-deployment to create an on-demand backup
# and subsequent daily backups (via EventBridge or cron in ops phase)

ENVIRONMENT=${1:-production}
REGION=${AWS_REGION:-us-east-1}
RETENTION_DAYS=${2:-7}

echo "🔄 DynamoDB Backup Strategy for $ENVIRONMENT environment"
echo "Region: $REGION, Retention: $RETENTION_DAYS days"

# Get table names from CloudFormation stack outputs
STACK_NAME="nunti-backend-$ENVIRONMENT"
DEVICES_TABLE=$(aws cloudformation describe-stacks --stack-name "$STACK_NAME" --region "$REGION" --query 'Stacks[0].Outputs[?OutputKey==`DevicesTableName`].OutputValue' --output text)
MESSAGES_TABLE=$(aws cloudformation describe-stacks --stack-name "$STACK_NAME" --region "$REGION" --query 'Stacks[0].Outputs[?OutputKey==`MessagesTableName`].OutputValue' --output text)

if [ -z "$DEVICES_TABLE" ] || [ -z "$MESSAGES_TABLE" ]; then
  echo "❌ Error: Could not retrieve table names from CloudFormation stack $STACK_NAME"
  exit 1
fi

echo "📊 Tables to back up:"
echo "  - $DEVICES_TABLE"
echo "  - $MESSAGES_TABLE"

# Create on-demand backups for each table
for TABLE_NAME in "$DEVICES_TABLE" "$MESSAGES_TABLE"; do
  BACKUP_ID="backup-$(date +%s)"
  echo "🔄 Creating backup for $TABLE_NAME..."
  
  BACKUP_ARN=$(aws dynamodb create-backup \
    --table-name "$TABLE_NAME" \
    --backup-name "$BACKUP_ID" \
    --region "$REGION" \
    --query 'BackupDetails.BackupArn' \
    --output text)
  
  if [ -z "$BACKUP_ARN" ]; then
    echo "❌ Backup failed for $TABLE_NAME"
    exit 1
  fi
  
  echo "✅ Backup created: $BACKUP_ARN"
done

# List existing backups and warn if older than retention period
echo ""
echo "📋 Existing backups (PITR is enabled separately via template):"
aws dynamodb list-backups --region "$REGION" --query 'BackupSummaries[*].[BackupName, BackupStatus, BackupCreationDateTime]' --output table

echo ""
echo "✅ Backup strategy executed. Remember:"
echo "  - PITR is always enabled for production tables (automatic, no manual action needed)"
echo "  - On-demand backups are point-in-time snapshots (retained per policy)"
echo "  - To restore: use scripts/deploy/restore-from-backup.sh {table-name} {backup-id}"
echo "  - Schedule this script daily via: EventBridge rule or cron job (Phase 11)"
```

Create scripts/deploy/restore-from-backup.sh:

```bash
#!/bin/bash
set -e

# Restore DynamoDB Table from Backup
# Usage: ./restore-from-backup.sh {table-name} {backup-id} {target-table-name-optional}

TABLE_NAME=$1
BACKUP_ID=$2
TARGET_TABLE=${3:-"${TABLE_NAME}-restored-$(date +%s)"}
REGION=${AWS_REGION:-us-east-1}

if [ -z "$TABLE_NAME" ] || [ -z "$BACKUP_ID" ]; then
  echo "Usage: $0 {table-name} {backup-id} {target-table-name-optional}"
  echo "Example: $0 nunti-devices-production backup-1234567890 nunti-devices-restore-test"
  exit 1
fi

echo "🔄 Restoring $TABLE_NAME from backup $BACKUP_ID"
echo "Target table: $TARGET_TABLE"

# Find backup ARN matching the backup ID
BACKUP_ARN=$(aws dynamodb list-backups \
  --table-name "$TABLE_NAME" \
  --region "$REGION" \
  --query "BackupSummaries[?BackupName=='$BACKUP_ID'].BackupArn" \
  --output text)

if [ -z "$BACKUP_ARN" ]; then
  echo "❌ Backup not found: $BACKUP_ID for table $TABLE_NAME"
  exit 1
fi

echo "Restoring from ARN: $BACKUP_ARN"

# Restore table
aws dynamodb restore-table-from-backup \
  --target-table-name "$TARGET_TABLE" \
  --backup-arn "$BACKUP_ARN" \
  --region "$REGION"

echo "✅ Restore initiated. Target table: $TARGET_TABLE"
echo "⏳ Table will be available in 1-5 minutes. Check status:"
echo "  aws dynamodb describe-table --table-name $TARGET_TABLE --region $REGION"
```

# DynamoDB Backup and Restore Procedures

## Overview

Production DynamoDB tables have:
1. **Point-in-Time Recovery (PITR):** Always enabled, automatic, no configuration needed
   - Can recover to any point within the last 35 days
   - No impact on table performance or throughput
   
2. **On-Demand Backups:** Manual snapshots created daily
   - Scheduled via `scripts/deploy/backup-strategy.sh`
   - Retained per policy (default: 7 days, configurable)
   - Can be restored to new table without affecting production table

## Backup Status: Daily Execution

Backups are created daily post-deployment via:
- Manual: `./scripts/deploy/backup-strategy.sh production`
- Automated: EventBridge rule (Phase 11 - Operations Readiness)

To verify backup status:
\`\`\`bash
aws dynamodb list-backups \
  --table-name nunti-devices-production \
  --region us-east-1 \
  --query 'BackupSummaries[*].[BackupName, BackupStatus, BackupCreationDateTime]' \
  --output table
\`\`\`

## Point-in-Time Restore (PITR)

Restore any table to a specific point in the last 35 days without creating a backup first.

**Via AWS Console:**
1. DynamoDB → Tables → (select table) → Backups → Point-in-time recovery tab
2. Select restore time → Create restore
3. New table is created with suffix `-restored-{timestamp}`

**Via CLI:**
\`\`\`bash
aws dynamodb restore-table-to-point-in-time \
  --source-table-name nunti-devices-production \
  --target-table-name nunti-devices-production-restored-$(date +%s) \
  --restore-date-time $(date -u +%Y-%m-%dT%H:%M:%S.000Z --date="1 hour ago") \
  --region us-east-1
\`\`\`

## On-Demand Backup + Restore

### Create Backup

\`\`\`bash
./scripts/deploy/backup-strategy.sh production 7
\`\`\`

Output lists all backups for tables in the production stack.

### Restore from Backup

Restore a specific backup to a new table:

\`\`\`bash
./scripts/deploy/restore-from-backup.sh nunti-devices-production backup-1712234567 nunti-devices-test-restore
\`\`\`

**Important:** Restores create a NEW table (do not overwrite production). Operator must:
1. Verify restored data integrity in test table
2. Update application config to point to test table (if desired)
3. Delete old test table when done: `aws dynamodb delete-table --table-name nunti-devices-test-restore`

## Restore Verification Checklist

After restoring:
- [ ] Restored table has same schema as production (attributes, keys, indexes)
- [ ] Row count matches expected snapshot (select approximate row count from Console)
- [ ] Data sampling: Query a few known records to verify accuracy
- [ ] Timestamp check: Verify restored data is from intended time
- [ ] No errors in CloudWatch Logs during restore

## Disaster Recovery Scenario

If production table is corrupted or accidentally deleted:

1. **Immediate:** Stop all Lambda invocations (disable API Gateway integration or Scale Lambda to 0)
2. **Identify:** Determine corruption time (check CloudWatch logs)
3. **Restore:** Use PITR to restore to 1 minute before corruption time
   \`\`\`bash
   aws dynamodb restore-table-to-point-in-time \
     --source-table-name nunti-devices-production \
     --target-table-name nunti-devices-production-restored-$(date +%s) \
     --restore-date-time "2026-04-10T12:00:00.000Z" \
     --region us-east-1
   \`\`\`
4. **Verify:** Restore verification checklist (above)
5. **Swap:** Once verified:
   - Delete corrupted table (or rename via backup/restore)
   - Rename restored table to original production name
   - Resume Lambda invocations
6. **Analyze:** Determine root cause of corruption and document

## Backup Retention Policy

**Default:** 7 days (minimum for compliance)
**Maximum:** 35 days (AWS limit for on-demand backups)
**Schedule:** Daily (via Phase 11 EventBridge automation)

To adjust retention:
- Update `BackupRetentionDays` parameter in `deploy/params.production.json`
- Redeploy: `./scripts/deploy/deploy-stage.sh production`
- Manual cleanup: Delete backups older than retention period via AWS Console or CLI

## Cost Implications

- **PITR:** $0.20 per GB per month (included in DynamoDB on-demand pricing)
- **On-demand backups:** $0.10 per GB (billed separately, one-time cost per backup)
- **Storage:** Backups consume S3 storage; old backups should be deleted

Cost optimization: Reduce backup frequency or retention period if backup costs exceed acceptable thresholds (Phase 11).

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Application → DynamoDB | Lambda writes must be idempotent; backups are read-only recovery artifacts |
| Backup storage | Backups are encrypted at rest by AWS; accessible only via IAM permissions |
| Restore point | Historical data is considered sensitive; restore should go to isolated test table first |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-07.1-11 | Tampering | PITR/Backup Data | mitigate | "PITR and backups are managed by AWS DynamoDB service (not application code). Backups are immutable once created. Encryption at rest is default. Access controlled via IAM (Lambda role cannot delete backup tables)." |
| T-07.1-12 | Information Disclosure | Restored test table | mitigate | "Restore creates a new table (original remains unmodified). Operator must explicitly delete test restore tables. Document deletion steps to prevent data leakage." |
| T-07.1-13 | Denial of Service | Restore operation | accept | "Restore creates new table; not blocking production. Concurrent restores consume AWS account table limit (default 256/region). Accepted as ops phase concern (Phase 11)." |
| T-07.1-14 | Repudiation | Backup audit trail | mitigate | "AWS CloudTrail logs all create-backup and restore calls. Enable CloudTrail logging in Phase 11 for audit compliance." |
