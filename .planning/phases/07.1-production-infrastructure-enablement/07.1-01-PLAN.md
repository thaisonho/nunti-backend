```markdown
## Production Network Architecture

### VPC Isolation

Production uses a separate VPC (10.1.0.0/16) completely isolated from staging:
- **Public Subnet:** 10.1.0.0/24 (NAT Gateway, Internet Gateway)
- **Private Subnets:** 10.1.1.0/24 (Lambda), 10.1.2.0/24 (Lambda backup)
- **NAT Gateway:** Enables Lambda outbound internet access (for external API calls) without exposing Lambda to inbound internet routing
- **Availability Zones:** Private subnets span 2 AZs for resilience

### Security Group Rules (Least-Privilege)

**Production Lambda Security Group (sg-production-lambda-placeholder):**
- Ingress: HTTPS from API Gateway only (source: API security group or CloudFront)
- Egress: 
  - HTTPS (443) to DynamoDB security group (CIDR: 10.1.1.0/25)
  - HTTPS (443) to external APIs (0.0.0.0/0 — required for Cognito, external integrations)
  - DNS (53/UDP) to 0.0.0.0/0 (required for DNS resolution)

**Production DynamoDB Security Group (sg-production-dynamodb-placeholder):**
- Ingress: HTTPS (443) from ProductionLambdaSecurityGroup only
- Egress: None (DynamoDB is inbound-only)

**Production API Gateway Security Group (sg-production-api-placeholder):**
- Ingress: HTTPS (443) from CloudFront distribution (or specific CloudFront IP range)
- Purpose: Bridges public internet → Private Lambda compute via edge

### Network Data Flow

```
Internet (Users)
    ↓
Route53 health check + failover logic
    ↓
CloudFront CDN (caching, DDoS protection)
    ↓
API Gateway (staging: direct, production: behind security group)
    ↓
NAT Gateway (ensures Lambda egress routing control)
    ↓
Lambda (in ProductionPrivateSubnet1/2, ProductionLambdaSecurityGroup)
    ↓
DynamoDB (VPC endpoint or intra-VPC, ProductionDynamoDBSecurityGroup)
```

### Operator Steps for Production Deployment

1. Verify production VPC exists in AWS account (or will be created by SAM template)
2. Update GitHub Actions secret `PROD_VPC_ID` with actual VPC ID (if VPC pre-created)
3. Update `deploy/params.production.json` with actual security group IDs output from first template.yaml deployment
4. Run production deployment: `./scripts/deploy/deploy-stage.sh production`
5. Verify security group ingress/egress rules in AWS Console → VPC → Security Groups
6. Test: curl through CloudFront to API Gateway → should reach Lambda → should reach DynamoDB
```

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Internet → CloudFront | Untrusted client requests; CloudFront is DDoS mitigation layer |
| CloudFront → API Gateway | Internal AWS edge; CloudFront must be only API Gateway ingress (security group) |
| API Gateway → Lambda | Managed service boundary; Lambda receives only legitimate API requests |
| Lambda → DynamoDB | Lambda has scoped IAM permissions; DynamoDB is intra-VPC (no public access) |
| Lambda → External APIs | Lambda escapes VPC via NAT Gateway; no direct internet exposure |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-07.1-01 | Spoofing | API Gateway | mitigate | "Security group restricts API Gateway to CloudFront only (not direct internet). CloudFront validates client certificates implicitly. Enforce HTTPS-only API Gateway (no HTTP)." |
| T-07.1-02 | Tampering | DynamoDB traffic | mitigate | "Lambda → DynamoDB uses TLS 1.2+. Security group restricts DynamoDB ingress to Lambda security group IP range only (10.1.1.0/25). No public DynamoDB endpoint." |
| T-07.1-03 | Tampering | Lambda process | accept | "Lambda runs in managed AWS environment (FaaS); tampering risk is AWS responsibility. Runtime IAM is least-privilege (no administrative permissions). Accept residual AWS infrastructure risk." |
| T-07.1-04 | Repudiation | Network egress | accept | "All VPC management and security group logs can be enabled via VPC Flow Logs (AWS CloudWatch). Deferred to Phase 11 (Operations Readiness). Accept logs-disabled for now." |
| T-07.1-05 | Information Disclosure | Secrets in params | mitigate | "Sensitive values (Cognito IDs, DB passwords) are injected via GitHub Actions secrets at deploy time. params.production.json contains only placeholders. Never commit real secrets to version control." |
| T-07.1-06 | Information Disclosure | NAT Gateway source IP | accept | "NAT Gateway provides a single source IP for Lambda egress, enabling operator audit. No mitigation needed; this is security benefit." |
| T-07.1-07 | Denial of Service | CloudFront | mitigate | "CloudFront provides AWS Shield Standard (free DDoS protection). Advanced threat patterns deferred to Phase 11. Documented under cost-conscious resilience tradeoff." |
| T-07.1-08 | Denial of Service | Lambda concurrency | mitigate | "Lambda concurrency is managed by AWS per-account/region limits. Burst protection is Phase 8 (Realtime Reliability). For now, document that uncontrolled Lambda invocation could exhaust compute capacity. Phase 8 will add bounded retry logic." |
| T-07.1-09 | Authorization | Security Group bypass | mitigate | "Security groups are AWS native controls, not application-layer. Enforcement is AWS responsibility. Verify security group rules in AWS Console post-deployment. Audit via gsd-secure-phase." |
| T-07.1-10 | Elevation of Privilege | IAM role trust | mitigate | "Lambda IAM role trusts Lambda service only, not arbitrary principals. OIDC deploy role is scoped to release workflows (Phase 6 OIDC setup). Least-privilege by default (no AdministratorAccess)." |

</threat_model>
